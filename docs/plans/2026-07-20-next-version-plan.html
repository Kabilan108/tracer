Tracer next version

Ship read-time tool-output filtering (#4) and native cross-host sync with annotation preservation (#5), plus a flake-versioned release pipeline: bump version in flake.nix → CI builds Linux binaries, tags, and publishes a GitHub release with the version embedded in the binary. Scope and non-goals

In scope

  • tracer get --tool-output=full|none|truncate:N — read-time filtering of <tool-use> blocks; archive files never modified.
  • tracer push <remote> + tracer receive — incremental cross-host transfer that merge-preserves receiver-side annotations.
  • Annotations (outcome/tag/untag) on opted-in additional roots, so receiver-side pipeline tags like wiki:compiled work.
  • tracer list tag-filter upgrades: repeatable --tag and ! negation (--tag '!wiki:compiled'), required by issue 5's payoff.
  • Version single-sourced in flake.nix; CI (currently none exists — greenfield .github/workflows/): test/lint on PR, and on version bump → build Linux binaries, tag vX.Y.Z, create release with binaries attached.
  • tracer skill — prints version-matched agent instructions (SKILL.md format with YAML frontmatter), mirroring pagebin skill, so agent skills can defer to the installed binary instead of hardcoding CLI knowledge that goes stale.

Non-goals

  • No change to the on-disk archive format — v2.1 markdown + YAML frontmatter stays as-is (filtering is read-time only).
  • No custom network protocol or SSH library — transport rides plain ssh (see Q3).
  • No bidirectional sync / pull; push-only, matching the current jacurutu → sietch topology.
  • No new Go dependencies anticipated: os/exec+ssh, archive/tar, crypto/sha256, and the existing modernc.org/sqlite state store cover everything.
  • Dotfiles changes (tracer-sync unit → tracer push wrapper, tracer-digest dropping seen.txt) are follow-ups in the dotfiles repo, noted but not part of this repo's work.

Current → target flow

Transcript consumption (issue #4)

Nowingest renders once (GenerateMarkdownWithMetadata → renderToolInfo, pkg/session/markdown.go:211); consumers regex-strip <tool-use> themselves; sessions ≈85% tool-output bytes
→
Targettracer get <id> --tool-output=none emits the archived markdown with tool payloads dropped to a one-line stub (tag + <summary> kept), or head-truncated with truncate:N. Tracer owns the format; consumers stop parsing it.

Key design consequence from the code: rendering happens once at ingest and get reads the finished .md. So filtering operates on the archived markdown itself — a depth-aware scan for <tool-use>…</tool-use> blocks — not a re-render from provider JSONL (source files may be gone, and re-rendering wouldn't work for sessions synced from other hosts). This also makes the filter work uniformly for every archived session ever written.

Cross-host sync (issue #5)

Nowdotfiles systemd unit rsyncs jacurutu → sietch /vault/userdata/tracer-ingest/jacurutu/; sietch reads via archive.additional_roots (read-only, enforced by mutationReadRoots, main.go:1145); rsync clobbers any receiver-side annotation; tracer-digest keeps parallel seen.txt
→
Targettracer push sietch transfers only new/changed sessions (SQLite cursor keyed on file hash), streams over ssh to tracer receive on sietch, which merge-preserves receiver annotations using the same ExtractAnnotations/ApplyAnnotations machinery sync/watch already uses locally (pkg/session/metadata.go:120, pkg/engine/engine.go:363). Receiver tags sessions wiki:compiled; state lives in frontmatter and survives re-pushes.

Release (new)

Nowversion duplicated: flake.nix:16 hardcodes 0.1.0, main.go:35 defaults "dev", just build passes no ldflags; no .github/ at all
→
Targetflake.nix is the single version source. Push to main that changes it → CI: tests → nix build → tag vX.Y.Z → GitHub release with Linux binaries; tracer version prints the release version because the existing ldflags injection (-X main.version) is reused everywhere, including just build.

Resolved Decisions

All six resolved by Kabilan on 2026-07-20. Option tables kept for the record; the decided option is marked on each card.

Q1 — truncate:N: lines or bytes? ✓ Decided: lines

N = lines per block, with a secondary hard cap (~8 KB/block) against pathological single lines.

OptionProsCons
Lines (recommended)Matches how you eyeball logs ("first 40 lines of build output"); stable across reflow; issue text says "first N lines/bytes" with lines firstA single pathological long line can still be huge (mitigate: secondary hard cap ~8 KB/block)
BytesDirectly bounds token costCuts mid-line/mid-word; harder to reason about when choosing N
Both (truncate:40 lines, truncate:2kb bytes)FlexibleTwo code paths + parsing for a flag with one real consumer today

Q2 — Include --turns=user,agent (conversation-only view) now? ✓ Decided: include in M2

OptionProsCons
Yes, in M2 (recommended)~Free once the block scanner exists (drop <tool-use> and <think> blocks); digest will want it for the cheapest passSlightly larger flag surface to stabilize
DeferIssue says --tool-output alone unblocks the pipeline; smallest M2Second release needed when digest wants it

Q3 — Push transport ✓ Decided: A — ssh + remote tracer receive

Confirmed detail: no persistent daemon on sietch. receive is invoked per-push by the sender over ssh, reads the tar stream from stdin, merges, writes, exits — the rsync-remote-side / git receive-pack model. No open port, no new systemd unit on the receiver.

OptionProsCons
A: ssh → remote tracer receive (recommended)Receiver-side merge is real code, not rsync tricks — reuses ApplyAnnotations; incremental via cursor; no new deps; systemd unit becomes one line; structured slog events on both endsRequires tracer installed on the receiver (true for sietch; M1's release pipe keeps it current); version-skew care across upgrades
B: smarter rsync wrapperNo receiver binary neededCannot merge-preserve receiver annotations — the core requirement; whole-file copy still clobbers. Fails issue 5
C: pure-Go SSH lib (golang.org/x/crypto/ssh)No ssh binary dependencyNew dependency + key/agent/known-hosts handling reimplemented; system ssh with your existing config is strictly simpler on this fleet

Q4 — Annotation merge semantics on re-push ✓ Decided: tags union · outcome receiver-wins · no sidecar

When the receiver's copy already has annotations and the sender pushes an updated file:

OptionProsCons
Tags: union · outcome: receiver wins if set (recommended)Receiver pipeline tags (wiki:compiled) survive; sender-side tag additions still arrive; matches issue's "keeps them" wording while being strictly more informativeA tag deliberately removed on the receiver reappears if the sender still has it (in practice sender never carries pipeline tags — document as a known limitation)
Receiver wins wholesale (tags + outcome, no union)Simplest mental model; exact "keeps them"Sender annotation improvements never propagate after first receiver touch
Sidecar overlay store on receiver (issue's alternative)Roots stay fully read-only; no merge logic in transferSplits annotation truth across frontmatter + DB; list/get/digest must consult both; not "inspectable in the file" — loses issue 5's stated payoff. Recommend rejecting (this also answers issue 5's "alternatively/additionally")

Q5 — Release binaries: nix-built x86_64 only, or cross-compiled matrix? ✓ Decided: nix-built linux-amd64 only

OptionProsCons
Nix build, linux-amd64 only (recommended)Bit-identical to what the flake/HM module deploys; CI validates vendorHash for free; CGO_ENABLED=0 → fully static, portable asset; flake is x86_64-linux-only today anywayNo arm64 asset (fleet is x86_64; add later by widening systems)
setup-go matrix (amd64 + arm64)Faster CI; arm64 nowRelease artifact built differently from the deployed path; version/ldflags logic duplicated outside the flake; vendorHash drift caught later

Q6 — Next version number? ✓ Decided: 0.2.0

Flake says 0.1.0; the markdown format banner says v2.1.0 (separate concern — format version stays untouched). This release is the largest feature set since the archive rework. Recommend 0.2.0 — stays semver-honest pre-1.0; the alternative is jumping to 2.2.0 to align binary and format versions, which couples two things that change at different rates. Also fine to split: M1 ships as 0.1.1 to prove the pipeline, features land in 0.2.0.

Release workflow design (M1 detail)

flake.nix becomes the single source of truth; everything else derives from it. Per Q6, this feature set ships as v0.2.0.

SurfaceChange
flake.nixKeep version = "X.Y.Z" (line 16) as the canonical value; existing buildPhase ldflags already inject it — unchanged
justfilebuild gains -ldflags "-X main.version=dev-$(git rev-parse --short HEAD)" so local binaries self-identify
ci.ymlon PR + push to main: gofmt -l (fail if nonempty), go vet ./..., go test ./..., nix build .#tracer (vendorHash gate)
release.ymlbelow
 
# .github/workflows/release.yml (sketch)
on:
  push: { branches: [main], paths: [flake.nix] }
  workflow_dispatch: {}
jobs:
  release:
    runs-on: ubuntu-latest
    permissions: { contents: write }
    steps:
      - uses: actions/checkout@v4
      - uses: DeterminateSystems/nix-installer-action@v14
      - uses: DeterminateSystems/magic-nix-cache-action@v8
      - id: v
        run: echo "version=$(nix eval --raw .#tracer.version)" >> "$GITHUB_OUTPUT"
      - name: Skip if already released
        run: git fetch --tags && (git rev-parse "v${{ steps.v.outputs.version }}" 2>/dev/null && exit 78 || true)
      - run: nix build .#tracer          # tests already gated by ci.yml on the same push
      - name: Package
        run: |
          install -D result/bin/tracer dist/tracer-linux-amd64
          (cd dist && sha256sum * > SHA256SUMS)
      - name: Tag + release
        env: { GH_TOKEN: '${{ github.token }}' }
        run: |
          git tag "v${{ steps.v.outputs.version }}" && git push origin "v${{ steps.v.outputs.version }}"
          gh release create "v${{ steps.v.outputs.version }}" dist/* --generate-notes
  • Trigger semantics: paths: [flake.nix] narrows the trigger; the tag-exists check makes it idempotent, so a flake.nix edit that doesn't change the version (e.g. vendorHash) is a clean no-op. workflow_dispatch covers manual re-runs.
  • Version verification: release job runs result/bin/tracer version and asserts it equals the flake version before publishing — closes the "binary has the version properly stored" requirement end-to-end.
  • Version extraction uses nix eval, not grep — immune to formatting changes in the flake.

Test strategy and observability

  • Unit (table-driven, per repo convention): filter_test.go (modes × nested tags × empty blocks × frontmatter preservation); metadataMatches tag matrix; pkg/transfer merge matrix + cursor diffing + path-traversal rejection.
  • Integration: push→receive end-to-end through a local pipe (no ssh required in CI): populate a fixture archive, push twice, tag on receiver between pushes, assert tag survives and only-changed files transfer.
  • VHS goldens (per cli-test conventions, extending tests/tapes/): get-filter.tape against tests/fixtures/archive; list negation tape; push-receive.tape against a tmp archive.
  • Observability: wide slog events per existing convention — one summary event per push (remote, scanned, transferred, bytes, skipped, failed, duration), one per receive, one per filtered get (mode, blocks_filtered, bytes_saved).
  • Gates: go vet ./... + gofmt after every change, go test ./... after every milestone (repo policy), CI enforces the same.

Definition of done

  • Bumping version in flake.nix on main produces, unattended: green tests → tagged vX.Y.Z → GitHub release with tracer-linux-amd64 + checksums, and that binary's tracer version prints exactly the flake version.
  • tracer get <id> --tool-output=none yields a conversation-narrative transcript with one-line tool stubs; truncate:N bounds each block; archive bytes untouched.
  • On sietch: tracer tag <id> wiki:compiled works on ingested sessions, and the tag survives the next tracer push from jacurutu.
  • tracer list --json --since 168h --tag '!wiki:compiled' returns the unprocessed set — seen.txt is deletable.
  • tracer skill prints accurate, version-matched instructions covering every command shipped in this release.
  • README.md, docs/configuration.md, and CHANGELOG.md updated (repo policy); issues #4 and #5 closable with links to the release.
